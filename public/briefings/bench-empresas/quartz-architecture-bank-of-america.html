1 Quartz — o que é, de onde veio

Quartz é a plataforma de tecnologia proprietária do Bank of America Merrill Lynch para banking e trading, em produção desde 2018 (Markets Media). Não é um produto comercial — é a infraestrutura tecnológica core do banco. É um caso clássico do que a indústria de IB chama de "Bank Python": uma escola de engenharia financeira que combina Python estendido, Object Database centralizado e monorepo gigantesco.

2 Os 5 pilares arquiteturais — o que torna Quartz diferente de Python convencional

Dev Python experiente não é dev Quartz. Os 5 pilares abaixo explicam porque a curva é vertical:

1
Object Database centralizado
Tudo é objeto Python persistente (Trade, Instrument, Curve, MarketData, RiskModel) versionado no Object DB. Source-of-truth única para todo o banco.
Impacto: não há ORM. Não há SQL no fluxo principal. O dev pensa em objetos vivos, não em tabelas — paradigma alheio à maioria dos devs Python web.
2
Monorepo Python único
Milhões de linhas de Python em uma única árvore de código. Todo o banco escreve no mesmo repo. Build massivo, testes integrados ao commit, code review centralizado.
Impacto: dependency hell vira dependency-by-design. PRs competem por "terreno". Onboarding técnico exige aprender a navegar uma base que é maior que o Linux kernel.
3
Hot-reload em produção
Código pode ser atualizado em processos rodando sem restart. Pricing engines, risk grids e trade lifecycle continuam vivos enquanto absorvem mudanças.
Impacto: o conceito de "deploy" web não existe. Mudanças propagam em segundos. Code de dev mal feito pode quebrar runtime de produção — disciplina extrema é mandatória.
4
Custom type system + Python estendido
Sistema de tipos próprio, anterior a typing do Python 3. Interpretador Python customizado com extensões em C++ (Cython/pybind11) para performance crítica.
Impacto: type hints convencionais não bastam. Há vocabulário próprio (Instrument, Curve, MarketData, RiskModel) que toma semanas para se internalizar.
5
Event-driven pricing
Tick de market data dispara re-cálculo automático de todos os derivativos dependentes. Grid in-memory distribuído em milhares de cores. Computação reativa em DAG.
Impacto: conceito de "função pura" é raro; quase tudo tem efeito colateral em cadeia. Dev precisa pensar em DAGs reativas, não em fluxo procedural.

3 Stack técnica (composição reconstituída a partir de vagas + papers)

BofA não publica documentação oficial. A reconstituição abaixo é baseada em vagas públicas no BoA Careers, papers do Calum Paterson ("The New Stack"), Markets Media (cobertura do go-live 2018) e analogia com Athena (JPM, mais documentado).

UI Layer
ReactJS (apps novas, padrão atual) · Tk/Win32 (legado em sunset) · Electron para desktop apps de traders · TypeScript onde o stack permite
Application Layer
Pricing tools · Risk reports (V@R, sensitivities, stress) · Regulatory (FRTB SA/IMA, IFRS9, BCBS239, Dodd-Frank) · Pre-trade analytics · PnL attribution · Trade lifecycle (booking, confirmation FpML, settlement)
Domain Layer
Instruments: FICC (IRS, OIS, basis swaps, FRAs, FX forwards, NDFs, options), Equities (cash, derivatives, structured), Credit, Commodities · Models: Black-Scholes, LIBOR Market Model, Hull-White, SABR, local vol, Heston · Curves: OIS, basis, credit, FX forward
Core Layer (Bank Python heart)
Object DB proprietário (hot in RAM, cold em disco distribuído) · Python estendido com type system customizado · Event bus (Kafka + proprietário) · Grid computing (in-memory, milhares de cores para risk run noturno)
Infrastructure Layer
Linux RHEL · C++ para pricing engines críticos (quants escrevem em Python, vai para C++ no runtime hot) · PostgreSQL / Oracle (persistência cold) · Kubernetes (serviços satélite, não core Object DB) · Hadoop / Spark (batch risk & regulatory)

Exemplo conceitual — código Bank Python idiomático

Reprodução pedagógica (não código real do BofA). Ilustra hot-reload, Object DB e event-driven pricing.

# Definição de instrumento — vive no Object DB, versionado, auditável class IRSwap(Instrument): # Type system custom: campos validados em build-time notional: Money fixed_rate: Rate floating_index: RateIndex # ex: USD_SOFR_3M start: Date maturity: Date pay_freq: Frequency def pv(self, market: MarketData) -> Money: # Event-driven: se a curve mudar, este pv é re-calculado automaticamente fixed_leg = market.discount_curve.pv_fixed(self) float_leg = market.forecast_curve.pv_floating(self) return fixed_leg - float_leg # Hot-reload: editar este arquivo recompila o módulo em runtime # Pricing engines vivos absorvem a nova lógica sem restart # Object DB versiona o objeto IRSwap — audit trail completo

5 Por que o ramp-up é tão lento — as 4 fricções estruturais

Vendors típicos (Bangalore, Manila, Mexico City) entregam dev com 30-50% de productivity nos primeiros 3 meses no Quartz. Isto é a norma — não exceção. Quatro forças combinam:

Fricção 1
Zero documentação pública

Tudo é tribal knowledge. Onboarding depende de seniores disponíveis (raros e caros). Não há "Quartz docs" ou Stack Overflow.

Fricção 2
Vocabulário financeiro denso

Dev "Python convencional" não sabe a diferença entre IRS e OIS, nem o que é negative basis trade, nem por que Greeks são caros de calcular. Glossário leva semanas.

Fricção 3
Idiom Bank Python

Programadores Python tradicionais escrevem código que não roda bem no Object DB — vazam memória, quebram hot-reload, criam tight coupling indesejado, ignoram event propagation.

Fricção 4
VDI restrito + ambiente air-gapped

Dev trabalha dentro do VDI do banco. Não consegue testar localmente, copiar código, ou usar IA online "padrão". Aprender por tentativa e erro é proibitivamente lento.

Tese central da Foursys

Se atacarmos as 4 fricções de uma vez — com academia pré-alocação (resolve fricções 1, 2, 3 antes do VDI) + AI Pair Programmer dentro do VDI (resolve fricção 4 sem violar compliance) — entregamos dev com 80-100% de productivity em 6 semanas. Isso é diferencial defensável vs. qualquer body shop do mundo.

7 Stack técnico do AI Pair Programmer Foursys — 6 componentes

O Pair Programmer Foursys roda dentro do VDI BofA sem exfiltração de código. Cada componente foi desenhado para resolver uma fricção específica do ramp-up, com privacy-by-design.

1. RAG financeiro privado
Embeddings de Hull (Options/Futures/Derivatives), BIS papers, ISDA defs, CFA curriculum, QuantLib docs. Busca semântica durante codificação — dev pergunta "como precifico FX barrier option?" e recebe contexto teórico + exemplo de código.
Stack: Vector DB local (pgvector/Qdrant) no VDI · Embeddings sentence-transformers ou OpenAI ada-002 · LangChain orchestration
2. Code-aware Q&A bot
"Explica esse módulo de pricing FX option" → resposta com contexto do codebase Quartz + financial reasoning. Aceita anexar trecho de código e devolve interpretação contextual.
Stack: LLM via Bedrock / Azure OpenAI com no-training guarantee · Streaming response · Logs auditáveis para compliance
3. AI Pair Programmer
Sugestão de código em tempo real, contextualizada com Bank Python idiom. Conhece padrões: hot-reload-safe, Object DB-aware, event-propagation correct.
Stack: Cursor Enterprise / GitHub Copilot Enterprise como base · .cursorrules Foursys customizado · Prompts pré-tuned para Quartz/Athena/SecDB
4. Test generation engine
Dada uma função em Quartz, gera unit tests + edge cases financeiros: zero coupon, deep OTM, vol → 0, barrier just touched, negative interest rates, FX inverted quote.
Stack: LLM + property-based testing (Hypothesis) · Mutation testing (mutmut) · Pytest integration
5. Documentation surfacer
Lê código Python, gera markdown documentation. Preenche o vácuo documental do Quartz — toda doc é tribal knowledge, então cada módulo lido pelo Surfacer fica permanentemente documentado.
Stack: LLM + AST parsing (libcst, ast) · MkDocs generation · Versionamento Git-style das docs geradas
6. Onboarding analytics dashboard
Para CIO/COO ver: tempo médio de ramp-up por dev, retention rate, defect rate por sprint, velocity vs. baseline GBS, NPS interno BofA do squad. Transparência total.
Stack: Grafana / PowerBI · Coletores no GitHub Enterprise / Jira · Visível ao cliente em real-time
